# くればぁ様への確認事項（2026-06-18）

前回（6/12）の MTG で合意した「販売価格の計算式」をシステムに組み込むにあたり、
計算式に入れる数字をどこから取るか、が確定していない箇所があります。
以下 4 点を確認させてください。

## 質問 1: 在庫表の「仕入値」列が複数あるが、販売価格の計算にはどれを使うか

販売価格の計算式（前回合意済み）:

```
販売価格 ＝（仕入値 ＋ 固定費）÷（1 − 粗利率）
```

この計算式の「仕入値」として、在庫表のどの列の数字を使えばよいかを確認したいです。
カテゴリごとに在庫表の列名が異なるため、それぞれお聞きします。

### メッシュの在庫表

メッシュの在庫表には「仕入値」に関する列が 2 つあります。

| 列名 | 在庫表での位置 |
|---|---|
| **仕入値(m)** | 左から 12 番目あたりの列 |
| **現行仕入値(m)** | さらに右の方にある列 |

前回の MTG で「在庫表の仕入れ値は入荷時点の金額（棚卸用）」「販売価格は最新の仕入れ値で計算する」とお聞きしました。

確認したいこと:
- 「仕入値(m)」が入荷時の金額（棚卸用）で、「現行仕入値(m)」が最新の仕入れ値、という理解で合っていますか？
- 合っている場合 → 販売価格の計算には「現行仕入値(m)」を使います
- 違う場合 → それぞれの列が何を表しているか教えてください

| **仕切り単価/m** | 左から 7 番目あたりの列 |

確認したいこと:
- この 2 つの列は何が違いますか？

## 質問 2: 計算式の「カット長」について

くればぁ様から以前いただいた計算式には「カット長」が含まれていました:

```
販売価格 ＝（仕入単価 × カット長 ＋ 固定費）÷（1 − 粗利率）
```

一方、前回（6/12）の MTG では、カット長を含まない形でお話しされていました:

```
販売価格 ＝（仕入値 ＋ 固定費）÷（1 − 粗利率）
```

ここでの「カット長」とは、EC サイトで販売する際の長さ（1m、2m、3m…）のことだと思いますが、確認させてください。

### 具体例で確認

たとえば、品番「PES/F」のメッシュで、仕入値が 100 円/m、固定費 6,000 円、粗利率 50% の場合:

**パターン A: カット長を計算に入れる場合**（EC で 3m として販売する商品の場合）

```
販売価格 ＝（100円 × 3m ＋ 6,000円）÷（1 − 0.5）＝ 12,600 円
```

→ EC サイトには「PES/F 3m: 12,600 円」と掲載する

**パターン B: 在庫表の仕入値がすでに「メートルあたり」なので、カット長は EC 品番ごとの掛け算で対応する場合**

```
1m あたりの販売価格 ＝（100円 ＋ 固定費を1mで按分）÷（1 − 0.5）
```

→ EC の 3m 品は単純に 3 倍する

確認したいこと:
- 実際にくればぁ様が EC の販売価格を決めるとき、パターン A と B のどちらに近い計算をされていますか？
- 「カット長」のデータはどこにありますか？（EC サイトの商品ページに掲載されている長さ＝カット長、という理解で合っていますか？）

## 質問 3: ネトロン・トリカルで代替商品を提案するときの基準

メッシュの場合、お客様から問い合わせがあった商品の在庫がゼロだったとき、「目開き」や「メッシュ数」が同じ・近い別商品を代替として提案する機能を作っています。

ネトロンとトリカルの在庫表には「目開き」「メッシュ数」の列がありません。

確認したいこと:
- ネトロン・トリカルで「この商品の在庫がないけど、代わりにこちらはどうですか」と提案する場合、何を基準に「近い商品」を探しますか？
- たとえば「品番が似ている」「幅が同じ」「カラーが同じ」など、実際の営業で使っている判断基準を教えてください

## 質問 4: 商品ごとに「仕入れ先」が分かるデータはあるか

前回（6/9）の MTG で、固定費（主に送料）を商品ごとに調整する際に「仕入れ先」と「幅」でフィルタリングできると便利、というお話がありました。

その際、在庫表に仕入れ先の色分けがあるとのことでしたが、古川に確認いただいている途中で終わっていました。

確認したいこと:
- 在庫表の色分け、または別のデータで、「この品番の仕入れ先は○○」という紐づけが分かる状態になっていますか？
- もし紐づけが分かるデータがあれば、システムに取り込んで、仕入れ先ごとに固定費をまとめて設定できる機能を作れます
